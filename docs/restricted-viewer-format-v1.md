# MD//WORKS Restricted Viewer Format v1 Revision 0.2

**Status:** Draft / Experimental
**Format identifier:** `mdworks-restricted-viewer`
**Format version:** `1`
**Initial implementation target:** MD//WORKS v1.5.7
**Specification revision:** `0.2`

## Compatibility-Critical Components

In Format v1, the following elements are primarily targeted for compatibility:

1. Method of identifying the format within HTML
2. Structure of the display data
3. Structure of the Markdown recovery data
4. Encryption methods and key derivation conditions
5. Verification methods during recovery

---

## 0. Conventions

The following terms are used in this specification:

* **MUST**: Requirements that a compatible implementation must strictly satisfy.
* **MUST NOT**: Actions that a compatible implementation must not perform.
* **SHOULD**: Requirements recommended to be followed unless there is a specific reason not to.
* **SHOULD NOT**: Actions recommended to be avoided unless there is a specific reason.
* **MAY**: Features that can be optionally adopted in an implementation.

---

## 1. Purpose

The MD//WORKS Restricted Viewer Format v1 is a format for saving and distributing Markdown documents as HTML files with the following properties:

* Can be viewed using only a web browser.
* In principle, requires no external communication.
* Can suppress text selection, copying, right-clicking, dragging, and printing.
* Does not require a password for viewing.
* Can restore the original Markdown to MD//WORKS using a recovery password.
* Can be generated by compatible tools other than MD//WORKS.

The viewing restrictions in this format are intended to suppress copying and printing through common operations.

The Restricted Viewer is not DRM. It does not completely prevent the acquisition of content via screenshots, screen recording, browser developer tools, or HTML parsing.

---

## 2. Scope

In Format v1, the following areas are targeted for compatibility.

### 2.1 Fixed Compatibility Areas

* Storage location of Restricted Viewer data within HTML
* Top-level JSON object
* Encrypted data for display
* Encrypted data for Markdown recovery
* Cryptographic algorithms and key derivation conditions
* Display payload
* Recovery payload
* Methods for identifying file format and version

### 2.2 Modifiable Implementation Areas

* Viewer color scheme and CSS
* Outline design
* Screen width, margins, and fonts
* Loading indicators and error messages
* Specific JavaScript implementations for copy suppression
* Mobile display layouts
* Accessibility aids
* Warning thresholds for file size

Even if the appearance or interface of the Viewer is changed, it is considered Format v1 compatible as long as the JSON structure and encryption format defined in this specification are maintained.

---

## 3. Terminology

### Restricted Viewer

Refers to the entire HTML document with viewing restrictions applied.

### Generator

Software that generates the Restricted Viewer HTML from Markdown, images, etc. MD//WORKS and `images2rHTML.py` fall under Generators.

### Reader

Software that restores the original Markdown from the Restricted Viewer. MD//WORKS falls under Readers.

### Display payload

JSON data containing the HTML body and title to be displayed on the browser.

### Recovery payload

JSON data containing the original Markdown, filename, and related metadata.

### Display box

An object where the Display payload is encrypted using AES-GCM. Because the display key is saved within the Viewer file, the Display box is not intended for confidentiality.

### Recovery box

An object where the Recovery payload is encrypted using a key derived from the recovery password.

### Recovery password

The password used to restore the original Markdown. It is not used during normal viewing.

## 4. File Requirements

### 4.1 Character Encoding

The entire file MUST be saved in UTF-8. The HTML must include:

```html
<meta charset="utf-8">

```

### 4.2 Recommended File Extension

The recommended filename format is:

```text
document.restricted.view.html

```

While Readers MAY target `.restricted.view.htm` for loading, Generators SHOULD generally use `.restricted.view.html`.

### 4.3 Standalone Operation

It is recommended that the Restricted Viewer be an HTML document capable of being opened as a local file (`file:///...`), requiring no external server connection to display the main text.

### 4.4 Performance and Size Considerations

Images embedded via Base64 will increase the file size. If image data is included in both payloads and then encrypted/Base64-encoded, the final HTML may be significantly larger than the original images. Format v1 does not define a universal maximum file size.

Generators SHOULD:

* Display estimated file sizes.
* Warn if the output becomes excessively large.
* Skip unprocessable images and continue processing.

Readers SHOULD protect against UI freezing caused by extremely long Markdown lines (like Base64 data URIs) and MAY omit auxiliary rendering processes (spell check, syntax highlighting) for such lines.

---

## 5. HTML Container

The Restricted Viewer HTML MUST contain exactly one `script` element like the following:

```html
<script id="mdworks-restricted-viewer-box" type="application/json">
{ ... }
</script>

```

Readers use this `id` to detect the Restricted Viewer data.

### 5.1 Required Element ID

`mdworks-restricted-viewer-box`

### 5.2 Required MIME Type

`application/json`

### 5.3 Recommended Metadata

Although optional, the following metadata is recommended for easy identification by tools:

```html
<meta name="mdworks-viewer-format" content="mdworks-restricted-viewer">
<meta name="mdworks-viewer-format-version" content="1">

```

---

## 6. Top-Level JSON Object

The `mdworks-restricted-viewer-box` element stores JSON in the following format:

```json
{
  "format": "mdworks-restricted-viewer",
  "formatVersion": 1,
  "display": { "...": "..." },
  "recovery": { "...": "..." }
}

```

### 6.1 Required Fields

| Field | Type | Required value |
| --- | --- | --- |
| `format` | string | `mdworks-restricted-viewer` |
| `formatVersion` | integer | `1` |
| `display` | object | Display box |
| `recovery` | object | Recovery box |

### 6.2 Version Handling

Readers process the file as Format v1 if `formatVersion` is `1`. Readers MUST NOT forcibly decrypt unsupported versions and SHOULD notify the user instead.

---

## 7. Display Box

The Display box is an object where the Display payload is encrypted using AES-GCM.

```json
{
  "alg": "AES-GCM",
  "keyFormat": "raw-base64",
  "key": "BASE64_ENCODED_32_BYTE_KEY",
  "iv": "BASE64_ENCODED_12_BYTE_IV",
  "data": "BASE64_ENCODED_CIPHERTEXT_AND_TAG"
}

```

### 7.1 Required Fields

| Field | Type | Requirement |
| --- | --- | --- |
| `alg` | string | `AES-GCM` |
| `keyFormat` | string | `raw-base64` |
| `key` | string | 32-byte key in standard Base64 |
| `iv` | string | 12-byte IV in standard Base64 |
| `data` | string | Ciphertext and auth tag in standard Base64 |

### 7.2 Security Characteristics

Because the key is stored in plaintext within the same file, the Display box cannot guarantee confidentiality, prevent developer tool extraction, or verify file authenticity. Its sole purpose is to avoid placing the display text as raw plaintext in the HTML source.

---

## 8. Display Payload

Decrypted Display box JSON:

```json
{
  "v": 1,
  "type": "mdworks-restricted-viewer-display",
  "title": "Presentation Title",
  "createdAt": "2026-06-22T12:34:56.789Z",
  "html": "<h2 id=\"slide-1\">Slide 1</h2><p>...</p>"
}

```

### 8.1 HTML Safety and XSS Protection

The `html` is inserted directly into the Viewer's DOM. Generators MUST sanitize this HTML before storing it. It MUST NOT include `<script>`, `<iframe>`, `<object>`, event attributes (`onclick`), or untrusted arbitrary HTML. Viewer implementations are recommended to perform an additional runtime sanitization right before DOM insertion.

---

## 9. Recovery Box

The Recovery box holds the Recovery payload encrypted via a password-derived key.

```json
{
  "v": 1,
  "label": "restricted-viewer-recovery",
  "alg": "AES-GCM",
  "kdf": "PBKDF2",
  "hash": "SHA-256",
  "iterations": 310000,
  "salt": "BASE64_ENCODED_16_BYTE_SALT",
  "iv": "BASE64_ENCODED_12_BYTE_IV",
  "data": "BASE64_ENCODED_CIPHERTEXT_AND_TAG"
}

```

---

## 10. Recovery Key Derivation

The AES key is derived using PBKDF2 with the following parameters:

* **Password encoding**: UTF-8
* **Hash**: SHA-256
* **Iterations**: 310,000
* **Salt length**: 16 bytes
* **Derived key length**: 32 bytes

**Password Processing:** The password string is passed to PBKDF2 exactly as-is. Implementations MUST NOT automatically trim whitespace, alter capitalization, or apply Unicode normalization to the password.

---

## 11. Recovery Payload

Decrypted Recovery box JSON:

```json
{
  "v": 1,
  "type": "mdworks-restricted-viewer-recovery",
  "title": "presentation.md",
  "filename": "presentation.md",
  "createdAt": "2026-06-22T12:34:56.789Z",
  "format": "mdworks-restricted-viewer",
  "markdown": "# Presentation\n\n..."
}

```

### 11.1 Markdown Encoding and Recovery Consistency

The `markdown` is stored as a Unicode string. Generators SHOULD retain the original Markdown content without automatically applying Unicode normalization. Readers MAY apply their own normalization (e.g., NFC) when importing the recovered Markdown into their editor environment.
